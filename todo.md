# 下一步实施清单

这份清单聚焦在 `AI 数据分析` 产品，而不是泛化聊天产品。目标是把当前项目从“能分析 CSV/Excel 的 Agent”继续收口成“可解释、可验证、可持续演进的数据分析助手”。

默认原则：

- 不做历史兼容包袱
- 本地开发和演示一律走 Docker
- 优先减少后续技术债，而不是先堆功能
- 先把产品边界说清，再补实现
- 优先做能提升答案可信度和复现性的能力

## 领域对齐后的产品判断

结合 2025-2026 年主流 AI 数据分析产品演进，当前赛道的核心已经不是单次 NL2SQL，而是以下五类能力：

- 语义层/知识层：字段语义、指标定义、候选关联关系、同义词、业务口径不能完全靠模型临场猜
- 可信答案：答案要能回看来源、SQL/Python 过程、使用了哪些表、做了哪些假设
- 多引擎协同：聚合/筛选/关联优先走 SQL，统计建模/复杂清洗/预测再切 Python
- 长任务稳定性：分析阶段和写报告阶段要分开，不让中间过程无限污染上下文
- benchmark/eval：没有稳定测例和回归指标，就无法判断 agent 是否真的变强

本产品当前阶段的明确边界：

- 核心场景：表格数据探索、多表分析、图表生成、书面报告
- 优先支持：CSV、Excel、本地上传数据
- 近期补齐：字段语义、关系发现、证据链、结果复核、benchmark
- 暂不追求：泛企业知识库问答、超重 BI 平台替代、过早微服务化

## P0：AI 数据分析产品最先补齐的能力

### 1. 数据语义层与导入元数据收口

目标：

- 让 agent 在分析前先拿到“数据语义摘要”，而不是只看到物理表结构
- 当字段名、表名、口径和关系不清时，先做小样本预分析，再让用户确认

需求：

- 导入时记录并持久化：
  - 表名
  - 原始文件名
  - 导入时间
  - 行数/列数
  - 字段类型推断结果
  - 非空率
  - 去重值数量估计
  - 代表性样本值
- 对字段生成可缓存的语义摘要：
  - 字段别名建议
  - 字段含义猜测
  - 时间/金额/比例/类别等角色标签
  - 可能的主键/外键候选
- 增加小样本预分析流程：
  - 只读取少量样本行
  - 结合列统计做字段语义猜测
  - 产出候选字段别名、指标含义、时间列、主键候选
  - 对低置信度项进入待确认状态
- 支持同一 session 下多表候选关系发现：
  - 列名相似度
  - 唯一值分布
  - 样本值重合度
  - 类型兼容性
- 给 agent 暴露统一的 schema context，而不是每次临时拼 prompt

验收标准：

- 同一批上传的多表数据，agent 能先读取语义摘要再决定分析策略
- 对高概率 join key 能给出候选关系和置信度
- 对低置信度关系不再静默假设
- 对列名含义不清、表关系不清的场景，系统会先展示候选理解再请求确认

### 2. 可信答案机制

目标：

- 让每个关键结论都能回答“你是怎么得出的”

需求：

- 每轮分析结果都要记录并可展示：
  - 使用了哪些表
  - 使用了哪些字段
  - 执行了哪些 SQL
  - 是否调用了 Python
  - 关键中间结果摘要
- 报告中的结论段落要能关联回：
  - 数据来源
  - 图表
  - SQL/Python 证据
- 对存在歧义的问题必须显式中断并确认：
  - 指标口径不清
  - 时间范围不清
  - join 关系不清
  - 维度含义不清
- 为后续 UI 预留“查看分析过程 / 查看 SQL / 查看假设”的展示结构

验收标准：

- 任何核心图表和结论都能追溯到具体数据来源和执行步骤
- 模型不再在歧义口径下直接输出看似完整的确定性答案

### 3. SQL 优先、Python 升级的执行路由

目标：

- 把“工具调用”收口成稳定的分析策略，而不是随机试错

需求：

- 明确默认路由：
  - 聚合、筛选、分组、排序、join、topN、同比环比优先走 SQL
  - 统计检验、时间序列预测、复杂清洗、异常检测优先走 Python
- 让 agent 输出本轮策略判断：
  - 为什么先走 SQL
  - 为什么需要 Python
  - 是否只是为了格式化结果而额外调用 Python
- 约束 Python 使用边界：
  - 默认只读分析库
  - 不允许无边界读全量大表到内存
  - 大数据量先做 SQL 下推再进 Python
- 为后续高级分析预留能力标签：
  - forecast
  - segmentation
  - anomaly detection
  - statistical summary

验收标准：

- 常规分析问题不再频繁绕到 Python
- Python 只在确有必要时介入，且过程可解释

### 4. 大规模数据导入与查询能力收口

目标：

- 先把“能支持多大数据规模”说清楚并做扎实

需求：

- CSV 导入改成真正流式，不再先把全量行读进内存
- Excel 明确产品上限，或替换成可控的流式读取路径
- 导入后自动补：
  - 行数统计
  - 基础列统计
  - 必要索引策略
  - 查询前的表级摘要
- 在 README 和产品说明里明确：
  - 单文件推荐上限
  - 单表推荐行数
  - Python 分析建议上限
  - 哪些场景会降级或拒绝执行

验收标准：

- 百万行级 CSV 导入不会因为全量读内存直接失稳
- 大表聚合查询能力有明确边界且和产品承诺一致

## P1：产品可用性与交互稳定性

### 5. Agent 上下文按阶段拆分

目标：

- 控制 prompt 体积，降低长任务后半程漂移

需求：

- 把长任务拆成至少两个阶段：
  - analysis phase
  - report phase
- 明确阶段切换时保留内容：
  - 数据摘要
  - 已确认口径
  - 已产出的关键发现
- 明确阶段切换时丢弃内容：
  - 大段原始 SQL 结果
  - 图表参数草稿
  - 重复的推理碎片

验收标准：

- 长任务后半程的 prompt 体积和响应延迟明显下降
- 报告阶段不再反复携带大段分析中间结果

### 6. 运行链路串行化和状态边界收紧

目标：

- 消除 session/run 生命周期中的竞争条件

需求：

- 明确一个 session 同时只能有一个 active run
- 系统性检查 `cancel/reset/new run` 的竞争条件
- 避免旧 run 的 emitter 污染新 run
- 明确 handler、session、engine、report state 的状态归属

验收标准：

- 快速点击“开始/停止/重新开始”不会把状态搞乱
- 页面刷新恢复后不会串 run

### 7. 结果复核与人工确认交互

目标：

- 把“分析助手”做成 analyst-in-the-loop，而不是假装全自动正确
- 为后续 `ask_user` 中断确认能力预留标准化协议

需求：

- 增加需确认的交互节点：
  - 上传或导入完成后的首轮语义预分析结果确认
  - 首次检测到多个候选 join
  - 指标定义有多个可能口径
  - 时间粒度需要周/月/季口径选择
  - 预测类任务需要用户确认方法和区间
- 这些场景统一通过 `ask_user` 类能力触发，而不是让模型自由生成一段普通文本追问
- 为后续 UI 预留：
  - 接受该假设
  - 改用另一个字段/口径
  - 跳过该分析步骤
  - 把本次确认保存为该工作区的数据语义映射
- `ask_user` 至少要支持结构化字段：
  - `reason`
  - `question`
  - `options`
  - `required`
  - `scope`
  - `context_ref`

- 运行时状态要支持：
  - run 从 `running` 进入 `waiting_user_input`
  - 用户确认后恢复同一个 run，而不是新开 run
  - 可区分“用户拒绝回答”“用户跳过”“用户明确选择”

- 人机协作流程明确为：
  - 第一步：系统基于少量样本和列统计生成候选理解
  - 第二步：用户确认或修正字段含义、join key、指标口径
  - 第三步：系统把确认结果写回 session/workspace 级语义上下文
  - 第四步：后续分析优先使用确认结果，而不是重新猜

验收标准：

- 高风险推断场景下，系统会先确认再继续
- 用户可以看懂系统当前到底在等什么信息
- 用户确认过一次后，后续同类分析不会反复追问同一语义问题
- `ask_user` 触发后，系统不会因为没有用户输入而把 run 直接判定为失败

### 8. 生命周期清理

目标：

- 控制磁盘和运行期垃圾持续增长

需求：

- session 过期清理
- 元数据 SQLite 清理策略
- 分析 SQLite 清理策略
- 调试 trace 清理策略
- 临时文件和中间产物清理

验收标准：

- 本地运行一段时间后，磁盘不会无边界增长

## P1/P2：数据源与产品边界延伸

### 9. 数据源接入分层

目标：

- 先稳住“本地文件分析”，再向外部数据源扩展

需求：

- 明确数据源分层：
  - L1：本地上传 CSV/Excel
  - L2：对象存储中的文件
  - L3：关系型数据库只读接入
  - L4：云数仓/查询引擎接入
- 所有数据源统一抽象成：
  - source definition
  - schema summary
  - access capability
  - freshness metadata
- 数据源接入必须区分：
  - 快照导入分析
  - 直接只读查询

验收标准：

- 新增数据源时，不需要推翻现有 session/file/run 抽象
- 产品文案能清楚解释“上传导入”和“直接查询”的区别

### 10. 数据库连接支持

目标：

- 把“数据库连接”做成一等数据源，而不是把连接串伪装成文件

产品判断：

- 第一阶段不做“任意数据库全支持”
- MVP 先支持 `PostgreSQL`
- 第二阶段补 `MySQL`
- `ClickHouse / Snowflake / BigQuery` 放到后续，不和第一阶段混做

需求：

- 引入独立的数据源实体，而不是复用 `file`
  - `data_source`
  - `data_source_binding`
  - `connection_secret_ref`
- 数据源至少区分：
  - upload file
  - database connection
  - object storage file
- 数据库连接元数据至少包括：
  - source_id
  - workspace_id
  - source_type
  - driver
  - host / port / database
  - default schema
  - allowlist schema/table
  - auth mode
  - secret ref
  - last_tested_at
  - status
- 安全边界必须先明确：
  - 只支持只读账号
  - 禁止 DDL/DML
  - 必须有连接测试
  - 必须有限时、限行数、限并发
  - 默认只允许 allowlist 内的 schema/table
  - UI 不回显明文密码
- 查询模式明确分两类：
  - snapshot import：把选中的表/视图导入 session analysis SQLite，再继续分析
  - live query：直接对外部数据库执行只读查询
- MVP 先只做 `snapshot import`
  - 原因：更符合当前 session cache + SQLite 分析架构
  - 原因：更利于报告复现、结果追溯、长任务稳定性
- agent 需要新增可见上下文：
  - 当前有哪些可用数据源
  - 数据源类型
  - 已连接成功的数据库
  - 可导入的 schema/table 概览
  - 数据新鲜度和导入时间

验收标准：

- 用户可以在工作区里保存一个只读 PostgreSQL 连接
- 用户可以测试连接并浏览 allowlist 内的 schema/table
- 用户可以把指定表/视图导入当前 session，再走现有分析链路
- 报告能区分“来自上传文件”还是“来自数据库快照导入”

### 11. 存储 provider 继续抽象

目标：

- 保持文件与对象存储实现解耦

需求：

- 保持业务层只认 `file_id`
- 保持元数据层只认 `provider + storage_key`
- 在本地实现稳定后，补 `S3-compatible` provider
- 不把 MinIO 当成领域前提

验收标准：

- 后续切 MinIO / S3 / R2 / GCS 时，不需要改前端协议和工具层

## P2：benchmark、测试与工程质量

### 12. Benchmark 基线升级为产品评测体系

目标：

- benchmark 不只是跑通流程，而是衡量 AI 数据分析质量

需求：

- 建内部 benchmark 分层，至少覆盖：
  - SQL
  - Spreadsheet
  - Python
  - 图表
  - 报告
  - Agent 编排
  - 多表关系发现
  - 歧义追问/确认
- 每个 case 至少记录：
  - 输入数据
  - 用户问题
  - 期望答案类型
  - 参考 SQL/参考统计结论
  - 允许的答案弹性范围
- 评测指标至少包括：
  - 任务成功率
  - 错误恢复率
  - 首次回答正确率
  - 歧义场景主动追问率
  - 耗时
  - token 成本

验收标准：

- 每次核心 agent 改动后，都能看到质量、成本和耗时变化
- 能识别“回答更长了但其实更不准”的回归

### 13. 自动化测试补齐

目标：

- 让核心链路可持续回归

需求：

- 后端仓储测试
- service 层单元测试
- upload / run / report 的接口测试
- Docker 端到端冒烟测试
- 至少覆盖：
  - 登录
  - bootstrap
  - 上传文件
  - 开始分析
  - 保存报告
  - 刷新恢复

验收标准：

- 每次改动后可以自动验证核心链路

### 14. CI 基础能力

目标：

- 主干分支不再主要依赖手工回归

需求：

- `go test ./...`
- 前端构建
- Docker 构建
- benchmark smoke case
- 冒烟测试
- 后续再考虑更严格的 lint / e2e

验收标准：

- 主干分支改动不会靠手工回归兜底

## P3：中期架构演进

### 15. 元数据仓储保留 PostgreSQL 迁移接口

目标：

- 为未来多实例和更稳的元数据层做准备，但不提前迁移

需求：

- 当前运行时继续用 SQLite
- repository interface 保持不变
- 等业务稳定后，再补 PostgreSQL repository 实现
- 迁移目标只放在元数据层，不影响分析 SQLite

验收标准：

- 迁移 PG 时不需要推翻现有领域模型和 handler/service 边界

### 16. 长任务拆到 worker

目标：

- 为更长、更重、更并发的分析任务做准备

需求：

- 当前 run 仍在 WebSocket 进程内执行
- 后续把 run 调度和执行拆开
- API / WS 负责交互
- worker 负责分析执行
- Redis 或队列负责调度和取消信号

验收标准：

- 能支撑更长、更重、更并发的分析任务

## 建议执行顺序

### 第一批

- 数据语义层与导入元数据收口
- 可信答案机制
- SQL 优先、Python 升级的执行路由
- 大规模数据导入与查询能力收口
- 数据库连接支持（PostgreSQL snapshot import MVP）
- 运行链路串行化和状态边界收紧
- Benchmark 基线升级为产品评测体系

### 第二批

- Agent 上下文按阶段拆分
- 结果复核与人工确认交互
- 生命周期清理
- MySQL 连接支持
- 自动化测试补齐
- CI 基础能力

### 第三批

- 数据源接入分层
- 存储 provider 继续抽象
- live query 路径设计
- worker 化准备

### 第四批

- PostgreSQL repository 实现
- worker 化落地

## 不建议现在做的事

- 不要继续把产品理解成“自然语言转 SQL demo”
- 不要在字段语义和证据链没补齐之前，过度强调自动化结论生成
- 不要为了“未来可扩展”提前引入过重的微服务拆分
- 不要在 benchmark 和 trace 都没成形之前继续堆普通文本日志
- 不要把 MinIO 写死进业务层
- 不要在没有只读、安全边界和语义层之前贸然直连外部生产数据库

## 第一批可执行 issue 草案

以下拆分默认对应可以直接建 issue 的粒度，优先覆盖第一批工作。

### A. 数据语义层

1. 新增导入元数据表，持久化行数、列数、类型、非空率、样本值
2. 为 `describe_data` 增加可缓存 schema summary 输出，不再每次全量临时计算
3. 增加字段角色识别器，标注时间/金额/比例/类别/主键候选
4. 增加多表候选关系发现服务，输出 join key 候选与置信度
5. 增加小样本语义预分析器，输出字段别名、口径猜测和低置信度告警
6. 把 schema summary 和关系候选接入 agent file/source context

### B. 可信答案机制

7. 为每次工具调用保存结构化 trace，至少包含输入、输出摘要、成功失败、耗时
8. 为 SQL/Python 结果建立可回溯 evidence 对象，供报告章节引用
9. 为报告章节补“来源引用”结构，支持关联表、SQL、图表、分析步骤
10. 对 join 歧义、时间范围歧义、指标口径歧义增加中断确认协议
11. 设计 `ask_user` 协议，统一承载歧义确认、口径确认、join 确认和预测参数确认

### C. 执行路由

12. 在 agent 层增加 SQL vs Python 路由策略说明，要求显式给出选择原因
13. 收紧 `run_python` 边界，默认只读、限制输入规模、禁止无边界全表加载
14. 增加高级分析能力标签，为 forecast/anomaly/segmentation 预留执行入口

### D. 大规模数据与导入

15. 把 CSV 导入改成真正流式实现，消除全量读内存路径
16. 给 Excel 明确产品上限，并在超限时返回稳定错误而不是随机失稳
17. 导入后自动生成表级统计和必要索引
18. 在 README 中写清数据规模边界和 Python 分析建议上限

### E. 数据库连接 MVP

## 数据库连接的明确产品范围

为了避免后续需求漂移，数据库连接能力先明确如下：

- 第一阶段只支持工作区级共享连接，不做用户个人连接
- 第一阶段只支持只读分析，不支持写回数据库
- 第一阶段只支持把外部表/视图导入当前 session 分析库，不做跨库实时 join
- 第一阶段只支持 PostgreSQL
- 第二阶段补 MySQL
- 第三阶段再评估 Snowflake、BigQuery、ClickHouse
- 不支持要求开超级权限或关闭数据库安全策略的接入方案
- 当表名/列名/关系含义不清时，默认先走“小样本 AI 预分析 + 人工确认”，不直接硬分析
