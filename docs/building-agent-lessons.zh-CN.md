# 把智能体做成 Runtime：我们在 Agent 实践中的几条经验

做这个数据分析智能体的过程中，我们越来越清楚一件事：智能体系统真正难的部分，不是让模型“会调用工具”，而是想明白 runtime、工具、状态和模型判断之间的边界。

如果边界划错了，系统很快就会滑回一种熟悉但低效的形态: 表面上是 agent，实际仍然是 workflow engine，只是把固定流程换成了几段更长的 prompt。下面这些经验，基本都是我们在这个项目里反复踩坑之后收敛出来的。

## 1. 不要把 Agent 写成隐形工作流

最容易犯的错误，是提前替模型决定路径。比如默认它一定要先读表，再看 schema，再写 SQL，再画图，再写报告，最后收尾。这个顺序在简单任务里看起来合理，但一旦任务变复杂，问题就会暴露出来。

真实的 agent 任务往往不是线性的。有时模型应该先摸清楚几个表的关系，再决定分析口径；有时应该先记录关键事实，再继续下探；有时一个子问题应该被拆出去单独处理。如果 runtime 预先规定“先做什么、后做什么”，模型的判断空间会越来越小，系统最后只是在执行一条被伪装过的脚本。

我们后来刻意把系统职责收窄到四件事：给目标、给工具、给状态、给边界。至于下一步怎么走，尽量由模型自己判断。这一点不是理念口号，而是智能体能否适应复杂任务的前提。

## 2. 系统提供状态，不替模型下判断

智能体系统很容易走向另一种过度设计: 模型一旦出错，runtime 就开始加“善意提示”，告诉它下一步应该做什么、哪里证据不足、现在适合委派还是适合收尾。

这种做法短期确实能提升一点稳定性，但长期代价很高。判断一旦被搬进 runtime，系统就会越来越像“代码在思考，模型在执行”。

我们的收敛方向是，系统可以暴露事实状态，但不替模型做裁决。比如：

- 工作记忆里目前保存了哪些事实
- 目标树里还有哪些分支未闭环
- 报告状态里有哪些 block、chart 和引用关系
- 当前上下文是不是已经开始膨胀

这些信息都应该作为可观察状态存在，而不是由系统翻译成“你现在应该先做 A，再做 B”。Judge 这件事，最好始终留在模型侧。

## 3. 观察工具比行动工具更关键

很多人做 agent 时，首先想到的是一批 action tool: 查数据库、跑 SQL、执行 Python、生成图表、写报告。这当然重要，但真正决定 agent 上限的，往往是 observation tool。

在我们的实践里，`state_memory_inspect`、`state_goal_inspect`、`state_report_inspect` 这类工具的价值非常高。它们的作用不是替模型分析，而是让模型能在需要的时候自己去看:

- 我已经确认了哪些口径和结论
- 当前还有哪些目标没有完成
- 报告结构是否已经失衡
- 图表和正文之间的引用是否完整

这类工具有一个非常重要的副作用: 它们能减少隐式指导。如果模型需要状态，就显式调用工具去拿；如果不需要，runtime 就不要主动塞给它。这样系统既保留了自主性，也减少了上下文污染。

## 4. 结构化状态应该是可选脚手架，不是强制思维路径

我们做了 working memory、subgoal tree、report block tree 这些结构化中间状态，但一个很重要的教训是: 这些结构存在，不代表模型必须按它们来思考。

例如，`goal_manage` 更适合作为一个可选 scratchpad。模型可以用它记录当前拆出来的目标，可以标记完成，也可以标记放弃；但它不应该变成一个强制性的 planning phase。同样，报告 block tree 是最终产物的组织方式，不应该反过来要求模型必须先列大纲、再逐段填充、最后才能继续分析。

结构化状态的价值，是让运行时更可观察、更可恢复、更容易校验；不是把模型重新驯化成一个阶段机。只要结构开始反过来规定推理顺序，agent 就已经在退化。

## 5. Guardrail 要薄，但必须足够硬

智能体不是不能加约束，而是不能把约束做成流程编排。

我们在这个项目里最明确的一个实践是: guardrail 只拦坏结果，不规定正常路径。比如在 `report_finalize` 这一步，我们会检查两类问题:

- 目标树里是否还存在未闭环的 active branch
- 报告状态本身是否非法，比如 block 重复、图表引用缺失、chart block 没有 caption

只要这些条件不满足，最终收尾就会被拒绝。这里的重点不在于告诉模型“你下一步应该补哪一段”，而在于明确系统不能落地一个坏的最终状态。

这类 guardrail 是必要的，因为它保护的是结果边界；但它也必须克制，因为一旦开始替模型规划过程，runtime 就会重新长成 workflow。

## 6. 长任务的核心问题通常不是能力，而是上下文失控

真正把 agent 跑长之后，我们遇到的最大问题之一，并不是模型不会分析，而是上下文越来越臃肿。

图表工具会带来很长的参数，报告工具会回灌大段正文，工具结果一旦原样保留，后半程 prompt 很容易被历史淹没。结果就是成本上升、延迟变高、有效信息占比下降，模型在最后几轮更容易漂移。

所以我们后来把很多精力放在上下文治理上，而不是继续堆 prompt 规则。几个有效做法是:

- 对历史工具调用做压缩，只保留后续推理真正需要的摘要
- 让模型把关键事实主动写入 working memory，而不是反复从长历史里回溯
- 当上下文达到阈值时，把更早的执行过程折叠成 digest
- 把 observation tool 设计成按需拉取，减少状态的重复注入

这类工作看起来不如“换更强模型”那么显眼，但它们才是长任务 agent 能否稳定运行的关键。

## 7. Delegate 是能力边界工具，不是表演型并行

子 agent 很容易被滥用。很多系统一旦支持 delegate，就倾向于把它当成“更高级的思考方式”，仿佛任务拆得越多，系统就越智能。

我们的经验正好相反。delegate 只有在两个条件同时满足时才真正有价值: 第一，子任务边界足够清晰；第二，子代理的工具权限能被明确裁剪。

也就是说，delegate 不是把不确定性扩散出去，而是把一个已经描述清楚的问题，交给一个受限执行单元。它的重点不是“多开几个 agent”，而是“让每个 agent 只在自己被允许的边界内工作”。

如果没有明确的任务说明、允许工具列表和状态隔离，delegate 很容易从能力放大器变成复杂度放大器。

## 8. Prompt 优化不能靠猜，要靠 trace 审计

Agent 做到后面，单看最终输出已经很难定位问题。你会看到“模型最后写得不稳”“它怎么又重复了”“为什么这一轮突然结束”，但如果没有 trace，这些判断几乎都停留在感觉层面。

所以我们后来把 trace 看得很重。不是为了做花哨的可视化，而是为了回答几个很实际的问题:

- 哪些工具参数正在持续撑大 prompt
- 哪类状态最容易被重复回灌
- 哪一轮开始出现明显的上下文劣化
- 某个 guardrail 被触发时，之前到底发生了什么

有了这些审计信息，很多“模型不稳定”的问题最后都会被还原成工程问题，比如工具 schema 太重、历史保留策略太粗、状态观察方式不够按需。对 agent 来说，trace 不是附属品，而是优化 runtime 的主要依据。

## 9. 工具描述要详细，但不要把建议写进契约

我们后来还修正了一个容易被忽略的误区: 不是所有“长 description”都会让系统变得不 agentic。

真正的问题不在于工具描述写得长，而在于它有没有偷偷替模型做判断。一个好的工具契约，完全可以写得很细，明确说明:

- 这个工具做什么
- 适用于什么场景
- 不适用于什么场景
- 参数分别表示什么
- 会读取哪些状态
- 会修改哪些状态
- 返回里有哪些事实
- 有哪些边界和失败条件

这类信息越清楚，模型越容易稳定使用工具。相反，真正应该避免的，是把“接下来建议怎么做”写进工具返回或工具描述，比如:

- `next_action`
- “现在应该先调 A，再调 B”
- “建议继续调用某个工具”

我们最终形成的共识是: 工具契约可以很详细，但它必须是 contract，而不是 workflow 提示器。

另外，UI 友好的展示摘要最好和事实载荷显式分开。我们现在更推荐统一使用 `ui_summary` 作为前端和运行日志消费的展示字段，而把真正给模型和工具链使用的信息保留在结构化事实字段里。这样可以避免 `summary_text` 一类字段同时承担“给人看”和“给模型压缩历史”两种职责，减少语义漂移。

## 结语

如果只总结一句话，那就是: 不要急着替模型规划路径，先把 agent runtime 的边界做对。

一个好的 agent 系统，应该擅长做这些事: 提供清晰目标、暴露可观察状态、设计边界明确的工具、允许模型自行判断、用薄 guardrail 拦住非法结果，并持续治理上下文复杂度。

当这些基础打稳之后，模型能力才会被真正释放出来。否则，无论 prompt 写得多漂亮，系统都很容易回到那种“看起来像 agent，实际还是 workflow”的老路上。
